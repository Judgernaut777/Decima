# Decima 0.3.0 — Local Daily Driver

> **EXPERIMENTAL — reference / prototype.** Decima 0.3 is not a finished product. Run it only on
> data you can afford to lose. It is a single-user, loopback-only local daemon; it is not a
> production, multi-user, or remotely-exposed service.

> **BACKUP FIRST.** The append-only Weft is your only canonical state. Take a backup with
> `decima-backup` before upgrading, before any experiment, and on a regular cadence. Backups
> exclude your signing keys by design — keep those keys backed up separately and privately.

Candidate package version: `0.3.0.dev0`. The `v0.3.0` tag is applied only after the operator
completes the pending gates in [`../RELEASE-READINESS.md`](../RELEASE-READINESS.md). The
independent audit verdict is **APPROVE WITH DOCUMENTED LIMITATIONS**
(`../release-evidence/audit/0.3-audit.md`).

## What Decima 0.3 is

Decima 0.3 turns the frozen `heartbeat/` reference implementation into a locally-hosted
daily-driver app. It ships a clean `decima/` package **extracted from and proven equivalent to**
`heartbeat/` (byte-equal via golden conformance fixtures), fronted by a **trusted Shell**: one
program you log into on `127.0.0.1` that connects you to your notes, tasks, projects, agents, and
approvals — with the system holding the authority so a rogue or prompt-injected agent still can't
act outside its capabilities.

## What is usable now

Through the rendered Shell today:

- **Notes** in trust zones, each with durable **provenance** (the Weft event ids that asserted it)
- **Tasks** and **projects**
- A **manual plan** lifecycle — start / pause (you drive it)
- The **agent inspector** — objective, principal, budgets, deadline, status
- The trusted **approval inbox** — gated terminate / revoke proposals defer to a reauth-gated approve
- **Capabilities**, **activity**, and per-item provenance

Operationally: clean install + first-run, `decima-doctor`, `decima-backup` / `decima-restore`
(state-root round-trip verified), and a systemd user service.

## Architecture preserved

- **Four verbs over one log** — ASSERT · RETRACT · INVOKE · ATTEST are the entire durable
  instruction set; there is no special grant/revoke.
- **The Weft is the sole truth** — every mutation routes through the kernel; projections (the
  Weave, boards, knowledge, activity) are disposable and rebuildable, never canonical. The only
  non-kernel database access is a read-only `SELECT` in the backup service.
- **No ambient authority** — a principal can do exactly what its capabilities permit; models
  *propose* and deterministic code *authorizes* + gates every effect. The kernel import-boundary
  guard enforces this at test time.

## Install prerequisites

- Linux (qualified on **aarch64 / arm64**), **Python ≥ 3.11**
- One third-party dependency: **PyNaCl** (real Ed25519 via libsodium); everything else is stdlib
- Optional service lifecycle: **systemd** (a user instance) and, to reproduce the container lane,
  **Docker**

Run the Shell directly:

```bash
python3 -m decima.shell.serve <weft.db>   # http://127.0.0.1:8973/
```

The pairing secret is written to a `0600` `.pairing-secret` file beside the Weft (only its path is
printed; pass `--print-secret` to echo the value on an interactive first run).

## Model-provider status

The **deterministic (offline) provider is the default** and the tested path. A live provider is
**opt-in** and requires an operator-supplied credential (env-var names in
`docs/operations/model-configuration.md`; no values in the repo). The non-live qualification
(routing + reason codes, budget block, privacy, failure/fallback, **secret redaction — 0 leaks**)
passes; the live smoke is **BLOCKED-pending-operator-credential** until an operator runs it with a
real key.

## Security boundaries

- **Strict same-origin CSP** — `script-src 'self'`, no `unsafe-inline`/`unsafe-eval`,
  `object-src 'none'`, `base-uri 'none'`; plus nosniff and frame-deny. Imported HTML/MD is proven
  non-executing; the frontend ships a dedicated `sanitize.js`.
- **Unauthenticated `/api/*` → 401**; every write route is `WRITE`/`REAUTH`.
- **Approval chrome** — hostile Approve buttons / banners / inline-JS are inert; only the real
  reauth-gated component can approve (re-present the pairing secret per call).
- **Worker isolation** — capability execution runs in an isolated child (chroot jail, no network,
  no credentials); worker-escape suite green.
- **Backups exclude signing keys**; the doctor report/export are scrubbed (no secret leaks).

## Known limitations & deferred features

- **The A–C Shell gap.** Source-grounded Q&A + clickable citations (`decima/capabilities/qa.py`)
  and the isolated coding workspace (`decima/capabilities/workspace.py`) exist as real,
  unit-tested **library modules that are NOT wired into the Shell** (no route, no screen). The
  Shell's plan lifecycle is **manual** — no model-generated plan and no agent forest spawned from
  the UI. Every security invariant those scenarios were meant to exercise is independently proven
  at its enforcing layer.
- **Single-threaded Shell server** — serves single-threaded to keep all Weft access on the one
  `sqlite3` thread; `/stream` returns a finite snapshot so requests serialize but never deadlock.
  The kernel fix (`weft.py` `check_same_thread=False` + lock) is filed for **0.3.1**.
- **Service lifecycle** is proven in a container (host systemd degraded), and the **live-provider
  smoke** is operator-gated.
- **Intentionally out of 0.3** (handoff §3.2): financial automation, live brokerage, full browser
  automation, mobile, replication/multi-device sync, and the eventual single Rust port.

## Upgrade expectations

This is the first coherent 0.3 release; there is no prior `decima/`-package release to migrate
from. The Weft is append-only and projections rebuild from it, so a projection schema bump is a
clean rebuild, not a data migration. **Always take a `decima-backup` before upgrading.** The kernel
and protocol are frozen for 0.3; 0.3.1 will pick up the single-threaded-server kernel fix and
Stage-2 kernel cleanup.
