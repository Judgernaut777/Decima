### The WS1 in-lane mitigation (single-threaded Shell loopback server) — classified

The Shell daemon (`decima/shell/serve.py::make_loopback_server`) serves **single-threaded** to
keep the single-connection `sqlite3` Weft on one thread. The auditor examined the DoS/correctness
angle: the only endpoint that could hold the sole thread is `/stream`, and
`events.py::sse_stream` returns a **finite snapshot list** of frames (`return [e.as_sse() for e
in self.since(cursor)]`) — it does **not** hold a long-lived connection. So requests serialize
but never deadlock the server. For a single-user local loopback daily driver this is invisible.
**Classification: Low / acceptable** for 0.3, with the real kernel fix (`weft.py`
`check_same_thread=False` + lock) correctly filed as a 0.3.1 follow-up in
`docs/release-evidence/browser/known-issues.md`. It is not a release-blocking correctness or DoS
risk in the shipped single-user posture.

## 8. Documentation-claim review — THE CRUX

**The Definition of Success requires scenarios A–C _through the Shell_.** The shipped Shell does
NOT deliver the aspirational A–C. Verified against the actual product (route table
`services/api/routes.py`, the frontend screens, and import graph):

| Aspirational scenario | Backend code exists? | Wired to Shell/API? | Shipped Shell actually does |
|---|---|---|---|
| **A** cross-source Q&A + clickable citations | **Yes** — `capabilities/qa.py` (cited, horizon-scoped, model-proposed) with unit tests | **No** — no `/qa`/`/ask` route; `qa.py` is imported by **nothing** in `decima/` | Knowledge = trust-zoned **notes** + durable **provenance** (Weft event ids). No Q&A engine, no clickable citation. |
| **B** model-generated plan → accept → agent hierarchy → resume → budget inspector | Partial | Partial | Create project, plan **start/pause**, agent **inspector**, gated terminate/revoke **proposals**. **No** model-generated plan, **no** accept step, **no** runtime spawning an agent forest from the UI; the bounded agent is a canonical-kernel harness precondition. |
| **C** isolated coding workspace, diff/test artifacts, no push | **Yes** — `capabilities/workspace.py` (mount → edit → run-in-worker → diff → apply → artifacts on Weft) with unit tests | **No** — imported only by `tests/capabilities/test_workspace.py`; **no** route, **no** Shell surface | **Nothing** — there is no coding-workspace surface in the Shell at all. |

So the "daily-driver capabilities" from commit `5351208` (Q&A, coding workspace) are **real,
tested library modules that are not reachable through the product's Shell**. The WS1 evidence
reports this honestly; the auditor independently confirms it via the route table and import graph.

**Is the A–C gap a Blocker/High, or an acceptable documented limitation?**

Decision: **acceptable documented limitation for a 0.3 "Local Daily Driver" scoped release —
PROVIDED the top-line docs stop implying the full experience.** Rationale:

- The 0.3 milestone is explicitly *"turns the reference into a locally-hosted daily-driver app"* with a trusted Shell + gated capabilities — not "ships the full A–C agent studio." The charter itself forbids expanding the capability surface during qualification.
- Every security invariant the scenarios were meant to *exercise* is independently proven at the layer that actually enforces it (worker isolation, approval gating, horizon-scoping, CSP, unauth-401), just not driven from the UI.
- README already has a strong, honest "What's real today vs. the vision" section framing the repo as a reference/prototype.

**But** the charter's *Definition of Success wording* ("A–C pass through the rendered Shell") is
**not literally met**, and a naive reader of `README.md` ("one program you log into that connects
you to your agents, your knowledge, and your work") could infer Q&A + coding + agent-forest are
in the Shell. That gap must be **documented at the top of the shipped docs**, not only buried in a
lane's evidence README. This is why the verdict is APPROVE-WITH-DOCUMENTED-LIMITATIONS rather than
a clean APPROVE.

### Other doc-claim inaccuracies found

- **`README.md` quickstart command is broken.** `python3 -m decima.shell.serve <weft.db>` runs
  with rc=0 but **does nothing** — `serve.py` has no `if __name__ == "__main__"` block, so the
  module imports and exits without starting a server. The working path is
  `deploy/decima-shell-server` (or calling `serve()`). A user following the README gets no Shell.
  → Finding **M-2**.
- **Stale test count.** `README.md` and `docs/RELEASE-READINESS.md` both say **"307 tests
  green"**; the actual suite is **323 passed + 5 skipped**. Understated, not inflated, but should
  be corrected for a release. → Finding **M-1** (documentation accuracy).
- **License metadata mismatch (informational).** `pyproject.toml` declares `license = "Proprietary"`
  and `Private :: Do Not Upload`; the surrounding Connect-ecosystem intent (per project memory) is
  Apache-2.0. Not release-blocking for a non-published `0.3.0.dev0`, but flag before any publish.

Capability labels (auditor's classification): trusted Shell (notes/tasks/projects/plans start-pause/
approvals/agents/activity) = **verified**; source-grounded Q&A + coding workspace =
**implemented-but-not-qualified** (library-only, not in Shell); model-generated plan / agent forest
from UI = **deferred**; live provider = **implemented, live path BLOCKED-pending-credential**;
deep systemd hardening on unprivileged user manager = **deferred/documented**. No wording claiming
full-OS-replacement, complete sandboxing on unsupported platforms, production remote access,
multi-device sync, self-forging, or live financial integrations survives as an unqualified claim
in the 0.3 surface.

## 10. Final recommendation

### APPROVE WITH DOCUMENTED LIMITATIONS

The candidate `2343b30` is a genuinely qualified, reproducible 0.3 release candidate for its
**actual scope**: a locally-hosted, capability-secured, single-user trusted Shell over an
append-only Weft, with clean install + first-run + backup/restore + doctor, strict CSP, unauth-401,
approval gating, worker isolation, and a bounded live-provider harness. Every automated gate the
auditor re-ran **passed** (323/5, ruff clean, wheel ships the frontend and serves 200 in
isolation, backup/restore state-root round-trip, doctor no-critical, browser 8/8 re-executed).
The frozen invariants hold byte-for-byte (`heartbeat/`, `decima/kernel/`, `protocol/`), the tree
is clean, and no `0.3` tag exists yet. No Blocker or High finding survives verification.

It is **not** a clean APPROVE because the charter's Definition-of-Success wording ("scenarios A–C
through the rendered Shell") is not literally met: source-grounded Q&A with citations (A) and the
coding workspace (C) are implemented and unit-tested but **not wired into the Shell**, and B is a
manual project/plan lifecycle with no model-generated plan or agent forest. This is an acceptable
scope for "Local Daily Driver 0.3" **only if the shipped top-level docs say so plainly**.

**Conditions to satisfy before applying the `0.3` tag:**

1. Land M-1, M-2, M-3 (correct the test count; fix or re-document the Shell run command; add a
   prominent "0.3 Shell scope" limitation covering the A–C gap). These are doc/DX fixes, not code.
2. Operator executes the live-provider smoke with a real credential (WS3 BLOCKED gate) and records
   `live-qualification.json`.
3. Re-run the full gate on the exact tag commit and confirm still 323/5 + heartbeat frozen + tree
   clean.

L-1, L-2, I-1, I-2 are follow-ups and do not block the tag. A rubber-stamp APPROVE that ignored
the A–C gap would have been a failed audit; this audit records the gap and scopes the release to
what the product actually does.

## 0.3 completion addendum — 2026-07-12 (candidate `9ec18a4`)

_This addendum does NOT rewrite the audit above; the §8 findings were accurate for candidate
`2343b30`. It records what changed on the later candidate `9ec18a4` (the P8 full-gate commit)._

**§8 finding M-3 — "scenarios A–C are not delivered through the Shell" — is RESOLVED.** The three
daily-driver workflows are now driven end-to-end through the rendered Shell and each is qualified
by a dedicated Playwright spec against the real backend (no canonical-store injection):

- **A — grounded Q&A** is wired through `decima/services/api/qa_service.py` +
  `js/screens/qa.js`, qualified by `tests/browser/specs/qa.spec.js`: import documents → choose a
  retrieval scope → ask → generated answer with citations that OPEN the real source passage;
  hostile imports stay inert DATA. (The library `capabilities/qa.py` the original audit found
  unwired is now composed by the service.)
- **B — model-planned durable agents**: `plan_service.py` + `js/screens/plans.js`, qualified by
  `planning.spec.js` — objective → routed model **proposal** → deterministic validation →
  `AcceptPlanProposal` mints durable Plan/Step/Agent → scheduler execution → pause/resume/cancel/
  gated terminate. This replaces the earlier "manual start/pause only" surface.
- **C — isolated coding workspace**: `workspace_service.py` + `js/screens/workspace.js`, qualified
  by `workspace.spec.js` + the containment suite — grant a repo root → bounded change → jailed,
  networkless `decima.workers` child → durable diff + test artifacts → restart recovery; no push,
  no credential, no network.

**Related resolutions on `9ec18a4`:**

- Finding **L-1** (pairing secret to the journal) and **M-2** (the no-op `serve` command) were
  already addressed in code before this candidate.
- The frozen invariants still hold byte-for-byte on `9ec18a4`: `heartbeat/` vs `3aa70d7`,
